# V1.2 Frontend Implementation Plan

> **For agentic workers:** executed via subagent-driven-development (fresh implementer per task, ECC react-reviewer per task, final whole-branch review).

**Goal:** Surface the merged V1.2 backend (customer progress updates + §649 cost-change approval) in the React UI: a "Kundeninfo" tab on the order detail page, a §649 cost-alert banner, a cost-change request + record-response flow, and an SMTP-status indicator.

**Architecture:** One new frontend API module (`api/customer-updates.ts`) fronting both endpoint families (customer-updates + cost-changes + projected-cost), with its own TS types (NOT in the shared `types.ts` — avoids the pre-existing `CustomerUpdateInput` name collision at `types.ts:112`). Four UI surfaces consume it. Order-detail tab state lives in `OrderContext` (`OrderTab` string-union, localStorage-keyed, NOT URL-driven) — a new `'kundeninfo'` member is added there. The cost-alert banner renders in the order header (always visible, not inside a tab). Cost-change form + history live in the existing `kosten` tab beside `CostBreakdownCard`.

**Tech Stack:** React 18 + TS + Vite, plain CSS with brand tokens, Vitest + happy-dom, `apiClient` (axios, Bearer interceptor), `useToast`/`useConfirm` contexts, `logError` (PII-stripping), `AuthenticatedImage` (blob-fetch for token-guarded images).

## Global Constraints

- **Permissions:** all V1.2 endpoints are ADMIN + GOLDSMITH only; VIEWER is excluded backend-side. The UI must degrade gracefully for VIEWER — hide send/create/record-response actions when the user lacks the permission (use the existing `useAuth()` role/permission check the way other tabs gate write actions). Read-only history may still show for VIEWER only if the GET is permitted (`CUSTOMER_UPDATE_VIEW`/`COST_CHANGE_VIEW` are NOT granted to VIEWER → VIEWER sees nothing; do not call the GETs for VIEWER).
- **PII / design-IP:** never log customer body text or photo references in plaintext — use `logError(context, err)` (strips axios request/response bodies) for every failed request. Photo selection for updates is explicit opt-in (`photo_ids`), never auto-attach all order photos.
- **Enum casing:** these backend enums serialize by value = **lowercase** (`progress|cost_change|ready_for_pickup|custom`, `draft|sent|send_failed`, `email|pdf_manual`, `draft|sent|approved|declined|superseded`, `email_reply|in_person|phone`, and record-response `approved|declined`). Frontend string-unions use the lowercase values verbatim. Each gets a German `Record<Union,string>` label map (mirror `getStatusLabel`/`StatusBadge`). Verify against one live API response before finalizing.
- **Money:** all amounts in `ProjectedCost` and `CostChangeRead.original_amount/new_amount` are **NET** (matches Quote.subtotal). Label them so a goldsmith reads them as netto (consistent with issue #28's pending gross/net copy decision — do NOT invent gross figures; show net + a "(netto)" qualifier).
- **No new backend endpoints.** SMTP status reuses `getEmailConfig()` from `api/admin.ts` (`EmailConfigRead`: `smtp_host, smtp_port, smtp_user, smtp_from, email_notifications_enabled, password_configured`). SMTP is "configured" when `email_notifications_enabled && smtp_host && password_configured`.
- **Async hygiene:** guard against out-of-order responses and stale state on unmount (mirror the `applyIfCurrent`/`actionLoading` pattern from `QuotesPage.tsx` merged in PR #30). Await error paths; show a toast on failure.
- Tests: Vitest, behavior-first (query by role/label/text), `vi.mock` the api module. `yarn tsc --noEmit` clean; `yarn vitest run <files>` green for touched files.

## Backend Contract (verbatim, from `api/routers/customer_updates.py` + `models/customer_update.py`)

| Method | Path (under `/api/v1`) | Perm | Body | Response |
|---|---|---|---|---|
| POST | `/orders/{order_id}/updates` | CUSTOMER_UPDATE_SEND | `CustomerUpdateCreate` | `CustomerUpdateRead` (201) |
| GET | `/orders/{order_id}/updates` | CUSTOMER_UPDATE_VIEW | — | `CustomerUpdateRead[]` |
| POST | `/updates/{update_id}/send` | CUSTOMER_UPDATE_SEND | — | `CustomerUpdateSendResult` |
| GET | `/updates/{update_id}/pdf` | CUSTOMER_UPDATE_VIEW | — | PDF stream |
| POST | `/updates/{update_id}/mark-delivered` | CUSTOMER_UPDATE_SEND | `MarkDeliveredRequest` | `CustomerUpdateRead` |
| POST | `/orders/{order_id}/cost-changes` | COST_CHANGE_MANAGE | `CostChangeCreate` | `CostChangeRead` (201) |
| GET | `/orders/{order_id}/cost-changes` | COST_CHANGE_VIEW | — | `CostChangeRead[]` |
| POST | `/cost-changes/{cost_change_id}/send` | COST_CHANGE_MANAGE | — | `CustomerUpdateSendResult` |
| POST | `/cost-changes/{cost_change_id}/record-response` | COST_CHANGE_MANAGE | `CostChangeRecordResponse` | `CostChangeRead` |
| GET | `/orders/{order_id}/projected-cost` | COST_CHANGE_VIEW | — | `ProjectedCost` |

**Schema fields (exact):**
- `CustomerUpdateCreate`: `kind` (enum), `subject?` (≤300, no CRLF), `body?` (≤20000), `photo_ids?` (≤20 UUID strings)
- `CustomerUpdateRead`: `id, order_id?, repair_job_id?, kind, subject?, body?, photo_ids[], cost_change_request_id?, token, status, sent_at?, sent_by, delivery_method?, created_at, updated_at`
- `CustomerUpdateSendResult`: `update: CustomerUpdateRead, delivered: bool, method?`
- `MarkDeliveredRequest`: `method: "pdf_manual"` (default)
- `CostChangeCreate`: `new_amount` (>0), `reason` (10–2000 chars stripped), `line_items?` (≤30, each `{label: string, amount: number, kind: "add"|"remove"|"change"}`)
- `CostChangeRead`: `id, order_id, quote_id?, original_amount, new_amount, delta_percent, reason, line_items[], status, response_method?, response_evidence?, responded_at?, recorded_by?, created_at, created_by, updated_at`
- `CostChangeRecordResponse`: `status: "approved"|"declined"`, `response_method: "email_reply"|"in_person"|"phone"`, `response_evidence` (5–2000 chars stripped)
- `ProjectedCost`: `material_cost, gemstone_cost, labor_minutes_billable, labor_cost, projected_total, quote_id?, quote_total?, delta_percent?, delta_abs?, over_threshold: bool, baseline_source?: "quote"|"approved_change"|null` (all NET)

- `recordCostChangeResponse(costChangeId: number, input: CostChangeRecordResponseInput): Promise<CostChange>` → POST `/cost-changes/${costChangeId}/record-response`
- `getProjectedCost(orderId: number): Promise<ProjectedCost>` → GET `/orders/${orderId}/projected-cost`

- Modify: `frontend/src/pages/OrderDetailPage.tsx` — add tab button (`👤 Kundeninfo`) + `{activeTab === 'kundeninfo' && <KundeninfoTab orderId={order.id} customerName={order.customer_name} />}`

**Behavior:** Two regions.
1. **History** — `customerUpdatesApi.listUpdates(orderId)` on mount; list newest-first with `kind` label, `status` badge (draft/sent/send_failed), `subject`, `created_at`, delivery method, and for `sent` a "PDF" button (`downloadUpdatePdf` → object-URL download, mirror `quotesApi.downloadPdf` usage). A `send_failed` or `draft` row shows a "Senden" and (for manual delivery) "Als übergeben markieren" action.
2. **Compose** — form: `kind` select (default `progress`), `subject` input (≤300), `body` textarea (≤20000), `<PhotoPicker>` for `photo_ids`. Show the send result honestly — `delivered:false` (no SMTP) → info toast "Als PDF erstellt — bitte manuell an den Kunden übergeben" and offer the PDF; `delivered:true` → success toast.

**Guards:** gate compose + send behind the write permission (`useAuth`), skip the GETs entirely for a user without view permission. `applyIfCurrent`-style guards + `actionLoading` for send/create. `logError` on every failure.

**Steps:** TDD — mock `customerUpdatesApi`, `getEmailConfig`, `PhotoPicker`; assert: history renders returned updates; compose "Erstellen & senden" calls `createUpdate` then `sendUpdate` and shows the delivered/PDF toast per result; SMTP-unconfigured shows the amber note; VIEWER (no perm) sees neither compose nor a GET call.
