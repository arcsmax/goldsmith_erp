# V1.2 Customer Updates & §649 Cost Approval — Backend Plan

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development. Spec: `docs/technical/specs/SPEC_V1_2_CUSTOMER_UPDATES_COST_APPROVAL.md` (governs). Frontend is a separate follow-up plan.

**Goal:** Progress emails with explicitly-selected photos, a projected-cost watcher implementing the §649 BGB Anzeigepflicht (15% / €150 thresholds), and CostChangeRequests with logged approval evidence. Email degrades to PDF-only when SMTP is unset.

**Architecture:** Assembly of existing machinery — `EmailService` (aiosmtplib+Jinja2, exists: `services/email_service.py`) gains a plain-text alternative part + photo attachments; `PDFService` (fpdf2, exists) gains a customer-update renderer; the cost watcher hooks the two established post-commit sites (`consume_material` after `_safe_publish_material_event`, `stop_time_entry` after `_check_and_publish_anomaly`) using the `_safe_publish` guarded fire-and-forget convention; scrubbing extends `SCRUBBABLE_FIELDS` via the existing `"order_id"` link.

## Global Constraints (binding — same as all V1.x plans)
- Classic Column style, SAEnum, idempotent migration_helpers, `transactional(db)` not nestable, typed exceptions raised OUTSIDE transactional when messages could carry user text, IDs-only in logs (customer PII NEVER — email addresses only via the existing `_anonymise_email`), `@require_permission` + `current_user` kwarg. Revision ids ≤32 chars (PG varchar limit!). JSON columns: `sa.null()` for SQL NULL. black/isort; new files mypy-clean via casts. Tests `poetry run pytest <path> -v` from repo root; PG suite in CI.
- Financial data (amounts, deltas) → ADMIN/GOLDSMITH; add `"customer-updates"` + `"cost-changes"` (or the single chosen route family) to `middleware/audit_logging.py` `_RESOURCE_ROUTES` — amounts are financial reads (V1.1 lesson).
- German customer-facing strings (emails, PDFs); code English.

### Task 3: CostWatchService + hooks

**Files:** `services/cost_watch_service.py`, edits in `services/metal_inventory_service.py` + `services/time_tracking_service.py` (one guarded call each), `tests/unit/test_cost_watch_service.py`.

- `CostWatchService.get_projected_cost(db, order_id) -> ProjectedCost`: material = SUM(MaterialUsage.cost_at_time) for the order (VERIFY column name; if `Order.material_cost_calculated` is cumulative use whichever is authoritative — read consume_material and decide, document); gemstones = SUM(Gemstone.cost*quantity); labor = billable-minutes rollup (copy the `case(Activity.is_billable)` aggregation from `time_tracking_service.get_summary:1009-1032`, filtered by order_id) / 60 × (order.hourly_rate or settings.DEFAULT_HOURLY_RATE). Quote reference: latest Quote for order with status in (SENT, APPROVED), fallback latest DRAFT; None-safe.
- `CostWatchService.check_order(db, order_id) -> Optional[ProjectedCost]`: computes; when quote exists and (delta_percent >= settings.COST_ALERT_THRESHOLD_PERCENT or delta_abs >= COST_ALERT_THRESHOLD_ABS_EUR) → create COST_ALERT Notification to ADMIN+GOLDSMITH users (title 'Kostenvoranschlag überschritten', message with order ref + percent — amounts OK, it's an internal financial notification; audit-logged surface) **deduped**: skip if an UNREAD COST_ALERT notification exists for related_order_id (adapt the deadline-warning dedup at notification_service.py:292-318 — unread-scoped, not day-scoped; once the goldsmith reads/acts, a further crossing may re-alert). Never raises (guarded like `_safe_publish`).
- Hooks: in `consume_material` AFTER `_safe_publish_material_event` and in `stop_time_entry` AFTER `_check_and_publish_anomaly` (both post-commit): `await CostWatchService.safe_check(db, order_id)` — a wrapper that try/excepts + logs, never raises, late-imports to avoid cycles. Skip when order_id is None.
- Endpoint (Task 5's router): `GET /orders/{id}/projected-cost`.
- Tests: rollup math (materials+gemstones+billable-only labor, rate fallback), threshold percent vs abs triggers, no-quote → no alert, dedup (second check no new notification while unread; after mark-read → re-alert), hook fire-and-forget (failure inside watcher doesn't break stop_time_entry — regression pattern from V1.1 Task 4 fixes).
