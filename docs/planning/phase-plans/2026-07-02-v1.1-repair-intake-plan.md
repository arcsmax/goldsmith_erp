# V1.1 Repair Photo-Intake Checklist — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development or superpowers:executing-plans. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Complete the last unbuilt V1.1 spec item (SPEC_V1_1 §Repair photo-intake checklist): a configurable intake checklist on RepairJob where each item is satisfied by an intake-phase photo or an explicit "nicht zutreffend"+reason — dispute protection mirroring insurance practice. Prerequisite folded in: **make repair photo upload real** (today the API stores client-sent `file_path` strings; the frontend sends ephemeral `blob:` URLs — no file ever reaches the server, and GDPR erasure of repair photos is a silent no-op).

**Architecture:** Mirror the proven consultation-photo machinery exactly: `repair_photo_storage` via the shared `image_validation` module, files under `PHOTO_STORAGE_PATH/repairs/{repair_id}/` with `thumbs/`; the existing `POST /repairs/{id}/photos` endpoint becomes multipart (its only consumer is the placeholder frontend being replaced — verified, no other callers); serve/thumbnail/delete endpoints added. `intake_checklist` JSON column on RepairJob seeded from a `list[str]` setting at creation; checklist items link to intake photos by id.

**Tech Stack:** Same as V1.1 throughout (FastAPI/SQLAlchemy async/Alembic + React/TS/Vitest). No new dependencies.

## Global Constraints (same as the V1.1 plans — binding)

- Backend: classic Column style, SAEnum, migration_helpers idempotent migrations, `transactional(db)` not nestable, business errors as typed/ValueError, structured logs with IDs only, `@require_permission` + `current_user` kwarg, black/isort/mypy on changed files. Tests: `poetry run pytest <path> -v` from repo root (no podman locally); PG suite via CI.
- Frontend: German UI, no new deps, string-union types, `logError` for ALL error logging (never raw err), `AuthenticatedImage` for ALL photo rendering, `vi.mock`-before-import tests, `npx tsc --noEmit` clean before every commit, 44px+ touch targets.
- Conventional commits, no attribution footer.

### Task 2 (backend): intake_checklist column + settings + endpoints

**Files:** Modify `src/goldsmith_erp/db/models.py` (RepairJob), `src/goldsmith_erp/core/config.py`, `src/goldsmith_erp/models/repair.py`, `src/goldsmith_erp/services/repair_service.py`, `src/goldsmith_erp/api/routers/repairs.py`; migration `20260703_r2_repair_intake_checklist.py` (chains after r1); Test `tests/unit/test_repair_intake_checklist.py`, extend integration.

- Settings: `REPAIR_INTAKE_CHECKLIST: list[str] = ["Krappen/Fassungen", "Pavé-Besatz", "Gravuren", "Punzen/Stempel", "Vorhandene Tragespuren und Schäden"]` (follows the BACKEND_CORS_ORIGINS list[str] precedent).
- Model: `intake_checklist = Column(JSON, nullable=True)` on RepairJob (+ idempotent add_column migration). Item shape (documented in the Pydantic schema, not enforced by DB): `{"key": str, "label": str, "status": "open"|"photo"|"na", "photo_id": int|None, "na_reason": str|None}`.
- Pydantic: `IntakeChecklistItem` (key/label required; status default "open"; na requires na_reason ≥3 chars — model_validator; photo status requires photo_id) and `IntakeChecklistUpdate(items: List[IntakeChecklistItem], max 20 items)`. `RepairJobRead` gains `intake_checklist: Optional[List[IntakeChecklistItem]]`.
- Service: `create_repair` seeds the checklist from settings (`[{key: slugified label, label, status: "open"} ...]`). New `RepairService.update_intake_checklist(db, repair_id, items) -> RepairJob`: validates every `photo_id` references an INTAKE-phase photo OF THIS repair (ValueError otherwise), replaces the column value. Endpoint `PUT /{repair_id}/intake-checklist` (`REPAIR_EDIT`), returns RepairJobRead.
- Tests: seeding on create; photo_id cross-repair rejection; na-without-reason 422; VIEWER 403 on PUT; checklist survives status transitions.
