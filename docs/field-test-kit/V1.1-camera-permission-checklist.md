# V1.1 Camera Permission — Manual Test Checklist (A8.3)

**Why this exists:** Slice 8 unit tests cover `NotAllowedError` at first
`getUserMedia` call, but they cannot simulate the iOS Safari
revoke-after-grant path — the OS kills the `MediaStream` asynchronously and
the browser only surfaces it as a track-ended event, not a fresh Promise
rejection. Lena (UX research, §3 "Camera permission re-denied") called
this out as the denied-state path most likely to regress silently, so we
test it on real hardware before ship.

**Devices required:**

- iPhone 13 or newer running iOS 17 (primary target — matches Thomas's
  shop iPad setup; the iPad lineage shares the iOS camera policy).
- Optional cross-check: Android (Pixel 6+, Chrome 120+).
- Optional cross-check: desktop Chrome / Firefox / Safari.

**Tester pre-requisites:**

- Build deployed to a secure origin (HTTPS or localhost). iOS refuses
  `getUserMedia` on http:// non-localhost.
- User logged in as a `GOLDSMITH` role (so the scanner FAB is visible).
- Volume on, tablet not muted (for the audio-feedback step).
- Clear browser data between test runs if repeating the grant-first path.

---

## 1. Grant-at-first-use (happy path sanity)

1. Open the app in Safari. Navigate to the scanner entry (ScanFab, Slice 10
   — for Slice 8 isolation: mount `QrCameraScanner` behind a debug route
   with `active={true}`).
2. On first tap of the FAB, iOS shows the system prompt
   *"<site> Would Like to Access the Camera"*.
3. Tap **"Allow"**.
4. **Expected:** live camera feed appears inside the viewport card, amber
   scan-region overlay visible, torch button visible only if device
   supports torch (iPhone main-camera yes, front-camera no).
5. Scan any QR code. **Expected:** one short vibration + OK beep, the
   `onScan` handler fires, the scanner pauses for ~800 ms then resumes.

**Pass criterion:** no denied-state card ever shown; `<=` 2 seconds from
tap to live feed.

---

## 2. Deny-at-first-use (NotAllowedError path)

1. Clear site data (Safari → Settings → Advanced → Website Data → remove
   the host).
2. Open app, navigate to scanner, tap FAB.
3. When the system prompt appears, tap **"Don't Allow"**.
4. **Expected UI, within 1 second:**
   - Amber-background card titled **"Kamera-Zugriff verweigert"**.
   - Large **"Manuell eingeben"** button (gold gradient).
   - Text input *already focused* (keyboard visible, cursor blinking).
     **CRITICAL:** no second tap required.
   - German help line: *"Einstellungen → Safari → Kamera → 'Erlauben'.
     Danach Seite neu laden."*
5. Type `ORDER:42` and tap the blue keyboard "return" / or tap the
   **"Senden"** button.
6. **Expected:** `onScan` fires with `('ORDER:42', 'manual')`. In Slice 10+
   context this opens the Quick Action Modal.

**Pass criterion:** time from "Don't Allow" tap to finger-on-keyboard for
manual entry is **<= 3 seconds** (A8.2 requirement).

---

## 3. Revoke-after-grant (the A8.3 scenario)

This is the path unit tests cannot cover.

### 3.a Setup

1. Complete section 1 ("Grant-at-first-use") successfully so the site has
   a stored camera grant.
2. Verify by tapping the FAB again — the camera should start **without**
   a system prompt.

### 3.b Revocation via iOS Settings (primary path)

3. With the camera feed live in the app, press Home (or swipe up) and
   navigate to **Settings → Safari → Camera**.
4. Change the permission for your site from **"Allow"** to **"Ask"** or
   **"Deny"**.
5. Return to Safari and the app (tab is still open).
6. **Expected behaviour within 2 seconds of returning:**
   - The live `<video>` feed freezes (track ends).
   - The component detects the stream going dead and transitions to the
     denied-state card. (NB: this currently depends on the
     `@yudiel/react-qr-scanner` library's internal error surfacing —
     if the library does NOT fire `onError` on track end, note this in
     the test report and file a follow-up; A8.3 is a *watched* behaviour,
     not a guarantee.)
   - Manual input autofocused, CTA visible.

### 3.c Revocation via site-Info flyout (secondary path)

7. Reset: grant camera again, confirm live feed.
8. With the feed live, tap the **`aA` icon in Safari's URL bar**, then
   **"Website Settings" → Camera → Deny**.
9. Tab auto-reloads. **Expected:** scanner opens in denied state directly
   (same as section 2).

### 3.d Revocation with the app backgrounded (bonus)

10. Grant camera, live feed visible.
11. Switch to another app for > 30 seconds (iOS may suspend Safari).
12. Return. **Expected:** camera resumes or the component re-requests
    permission. If a fresh system prompt appears, that's acceptable —
    note it in the test report for UX review, but it is NOT a regression.

**Pass criterion for section 3:** no state where the user sees a frozen
black video rectangle with no fallback. Denied UI must engage within 2 s
of track end.

---

## 4. Audio / haptic feedback

1. In the granted state, verify **OK beep** plays on a successful scan.
   Adjust volume via the in-overlay mute toggle — toggle persists across
   page reload (`localStorage.scan_audio_muted`).
2. Tap the mute button. Scan another code. **Expected:** no beep. Haptic
   (`navigator.vibrate`) still fires on iOS 13+.
3. Unmute. Scan a deliberately invalid / unreadable code (scribble on
   paper). **Expected:** error beep (descending tone), double-tap haptic.

**iOS quirk to accept, not flag:** on the very first scan after page load
the beep may not play because iOS requires a user gesture before
`HTMLMediaElement.play()`. This is browser policy, not our bug.

---

## 5. Matte vs. glossy label (A8.4 cross-check)

Not a permissions test, but run it while the hardware is in hand since
the scan success rate depends on label stock:

1. Print the same QR payload on glossy label stock and on matte label
   stock. Same laser printer, same size (20×20 mm target per spec §7).
2. Under workshop LED-4000K lighting, scan each 10 times from ~15 cm.
3. Document the success rate for both.

**Expected (per A8.4):** matte >= 95 % first-scan success; glossy
materially lower. Record the gap in the V1.1 ops handbook so field-test
only uses matte.
