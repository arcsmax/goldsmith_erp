# V1.1 ScanFab Position — iPad Bench Check (A10.1)

**Why this exists:** The ScanFab introduces a second floating-action-button
on top of the existing TimerWidget FAB. On Meister Thomas's iPad 9. Gen in
portrait orientation — the canonical bench setup — the two FABs plus the
"Foto aufnehmen" capture button must coexist without overlap or finger
collision. A10.1 ("capture a photo of Meister Thomas's iPad bench setup,
confirm FAB position doesn't collide") calls this out as a pre-ship gate
that is cheaper to verify now than to retrofit after field deployment.

**Related code:**

- `frontend/src/components/scanner/ScanFab.tsx`
- `frontend/src/styles/components/ScanFab.css` (see `.scan-fab--stacked`)
- `frontend/src/components/TimerWidget.tsx`
- `frontend/src/styles/components/TimerWidget.css`

**Z-index coordination (confirm in DevTools if anything regresses):**

| Layer | z-index | Owner |
|---|---|---|
| TimerWidget FAB | 1050 | `styles/components/TimerWidget.css` |
| ScanFab | 1060 | `styles/components/ScanFab.css` |
| ScanOverlay | 1500 | `styles/components/ScanOverlay.css` |

## Checks

### Baseline — no timer running

- [ ] ScanFab is visible bottom-right, inside the iPad safe area.
- [ ] TimerWidget "start timer" FAB is ALSO visible bottom-right,
      **at the same baseline**. They must NOT overlap each other.
      Expected: TimerWidget (`z-index: 1050`) sits at `bottom: 1.5rem`,
      ScanFab (`z-index: 1060`) stacks at `bottom: 1.5rem` too — same
      level, different z-order.

> **If they overlap visually:** the ScanFab CSS expects stacking only when a
> timer is running (`.scan-fab--stacked` class). If baseline layout collides,
> add a `.scan-fab--stacked` behaviour for the no-timer case too (cheap
> tweak to the `runningEntry` condition in `ScanFab.tsx`).

### Timer running — stacking behaviour

- [ ] Start a timer on any order.
- [ ] ScanFab moves UP by ~72px (offset = `--fab-bottom + 72px`).
- [ ] TimerWidget remains at the bottom-right baseline.
- [ ] Both FABs are fully tappable with a gloved thumb — neither partially
      occluded by the other.

### Thumb-reach ergonomics

- [ ] Hold iPad one-handed in the bench cradle, right thumb naturally
      resting at bottom-right. Both FABs fall inside the thumb-reach zone.
- [ ] No false taps when reaching for one FAB — the neighbouring button's
      hit area does not bleed into the targeted one.

### "Foto aufnehmen" capture button collision

- [ ] Open an order detail page that shows the camera capture button.
- [ ] Confirm the capture button does NOT overlap either FAB.
- [ ] If overlap occurs: note which page and which button, attach screenshot.

### ScanOverlay raise + close

- [ ] Tap ScanFab → full-viewport overlay raises, camera prompt appears.
- [ ] Deny camera → manual fallback renders within 3 seconds (A8.2).
- [ ] Close button (top-right) is reachable with a thumb without repositioning
      the iPad.
- [ ] Close → focus returns somewhere sensible (close button hidden, either
      ScanFab regains focus or focus returns to the triggering page region).

### Landscape orientation spot-check

- [ ] Rotate iPad to landscape. Confirm both FABs remain bottom-right
      (not pushed off-screen). Stacking behaviour should still apply when
      a timer is running.
