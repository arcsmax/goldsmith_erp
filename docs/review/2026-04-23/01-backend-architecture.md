# Review — Backend Architecture & API Design  (2026-04-23, HEAD 1feae6d)

## Scope reviewed

Targeted symbolic read of `src/goldsmith_erp/main.py`, `db/session.py`, every file under `core/`, the auth middleware, the full `api/routers/` tree (32 routers, sampled 10 in depth + cross-cutting grep across all), representative deep reads on `services/order_service.py`, `services/time_tracking_service.py`, `services/notification_service.py`, `services/user_service.py`, `services/photo_service.py`, `services/email_service.py`, `services/system_monitor.py`, and the Pydantic schemas in `models/user.py` + `models/order.py`. Cross-cutting greps for `selectinload`, `except Exception`, `db.commit`, `publish_event`, `response_model`, router-level SQL, sync I/O, and deprecated Pydantic v1 APIs. DB model schema (`db/models.py`) deliberately skipped — owned by the DB agent.

| P0 | `src/goldsmith_erp/api/deps.py:108-217` vs `src/goldsmith_erp/core/permissions.py:16-423` | **Two parallel RBAC systems coexist with divergent permission enums, divergent `ROLE_PERMISSIONS` maps, and divergent `require_permission` implementations** (one is a FastAPI `Depends` factory, the other a function decorator). 21 routers import from `core.permissions`, 5 import from `api.deps` (`customers.py:13`, `measurements.py:21`, `metal_types.py:23`, `metal_prices.py:27`, `metal_inventory.py:25`). This is a latent privilege-escalation / denial surface and a maintenance bomb. | Pick `core.permissions` as the single source of truth. Delete the `Permission` enum, `ROLE_PERMISSIONS`, `has_permission` and `require_permission` from `api/deps.py`; keep only `get_current_user` + `get_current_admin_user` there. Migrate the 5 routers to import from `core.permissions` and switch them to the decorator style (or convert the decorator to a `Depends` factory so both worlds use the same idiom). | P1 | `src/goldsmith_erp/services/time_tracking_service.py:99-139, 141-191, 363-390, 417-429, 431-456, 700-779, 781-868` | Seven mutating methods (`start_time_entry`, `stop_time_entry`, `create_time_entry`, `update_time_entry`, `delete_time_entry`, `add_interruption`, `patch_activity`, `log_interruption`) call bare `db.commit()` with no rollback path. If the activity-usage-counter update (`ActivityService.increment_usage`) or a subsequent `publish_event` raises, the primary row is already committed but the auxiliary state and the caller see an exception — leaving inconsistent state. `OrderService` uses `async with transactional(db)` for the same pattern; the inconsistency is cross-service. | Wrap all multi-step writes in `async with transactional(db)` (already available in `db/transaction.py`). Move the `increment_usage` / `update_average_duration` calls inside the same transaction, or explicitly document that they are idempotent post-commit side effects. |
| P1 | `src/goldsmith_erp/core/permissions.py:239-262, 284-307, 329-355, 377-397` | The `@require_permission`, `@require_any_permission`, `@require_all_permissions`, `@require_role` decorators extract `current_user` via `kwargs.get('current_user')`. This works only because FastAPI always passes `Depends()` results by keyword. It silently degrades to `None` (401) if a developer ever passes `current_user` positionally or renames the parameter, producing an opaque "Authentication required" for a logged-in admin. The pattern is also not compatible with `async def` routes that want to accept `current_user` under a different name (e.g. `user: User = Depends(get_current_user)`). | Convert `require_permission` to a FastAPI dependency factory (like the one in `api/deps.py`) so the check rides on the same DI pipeline as `get_current_user`. The decorator form can remain for non-route callers as a thin shim. |
| P1 | `src/goldsmith_erp/services/notification_service.py:86-120`, `services/order_service.py:385-468`, `services/time_tracking_service.py:870-954` | `NotificationService.create_notification` calls `await db.commit()` unconditionally on line 96, then `await EmailService.send_email` synchronously via `_try_send_customer_email` (line 106). Callers invoke this from inside other service methods (`_safe_publish_order_event`, `_safe_publish`). When called inside an `async with transactional(db)` block, the inner `commit()` terminates the outer transaction prematurely — any later work in that block will run in an auto-commit-per-statement mode. And synchronous SMTP in the request path can add seconds of latency per notification under a slow mail server. | Accept an explicit `commit: bool = True` flag, or (better) have the method only `flush()` and require the caller to own the transaction boundary. Fire the email via `BackgroundTasks` or a lightweight queue (asyncio task stored on `app.state` with a bounded-size deque) so SMTP never blocks the response. |
| P1 | `src/goldsmith_erp/api/routers/materials.py:359`, `api/routers/scrap_gold.py:175`, `services/photo_service.py:175, 188-103` | **Blocking sync I/O in async handlers**: `Path.write_bytes()` for multi-MB photo uploads, and PIL `Image.open/resize/save` for thumbnail generation run on the event loop thread. Under photo-upload load this stalls every other coroutine (including the WebSocket sender loops) for the duration of the disk/CPU burn. | Wrap filesystem writes in `anyio.to_thread.run_sync(...)` (or `asyncio.to_thread(...)` on Python 3.9+). Move PIL thumbnailing to a `run_in_executor` call. Alternatively use `aiofiles.open(..., 'wb')`. |
| P1 | `src/goldsmith_erp/main.py:213-217` | `@app.on_event("startup")` is deprecated since Starlette 0.26 / FastAPI 0.93 and carries no matching `shutdown` handler, so `system_monitor_loop()` runs forever without being cancelled on graceful shutdown — the task is destroyed pending, logging a noisy traceback, and its open DB/Redis connections are not released cleanly. | Migrate to the `lifespan` context-manager pattern documented in FastAPI 0.115: `async with asynccontextmanager` that spawns the task on enter and cancels it (with `asyncio.wait_for` timeout) on exit. |
| P1 | `src/goldsmith_erp/main.py:164-210` + `core/pubsub.py:60-75, 78-102` | WebSocket fan-out **opens one Redis pubsub connection per connected WebSocket client**. For the shared `order_updates` channel, N connected goldsmiths → N Redis subscriptions → N separate event-loop coroutines parsing the same message. Each WebSocket disconnect-without-closing-the-task (e.g. browser tab crash that TCP-resets rather than close-framing) leaks the coroutine for several seconds until Redis notices. There is also no connection cap, no per-user dedupe, and the in-flight `subscribe_task` is only cancelled on `WebSocketDisconnect` (not on server-side exceptions). | Introduce a single process-level `ConnectionManager` that owns one Redis subscriber per distinct channel, maintains `{channel: set[WebSocket]}`, and fans messages out in-process. Add a graceful shutdown hook to close all sockets. Cap concurrent connections per user (e.g. 5) to avoid abuse. |
| P1 | `src/goldsmith_erp/api/routers/metal_inventory.py:85-90, 137-142, 319-324, 368-373, 420-425, 474-479` | Broad `except Exception as e:` around the happy path of every router handler converts **any** service-layer error — including `AlloyMismatchError` subclasses not anticipated, Pydantic validation failures within the service, and even `asyncio.CancelledError` on Python <3.8 — into an opaque `HTTPException(400/500, detail=str(e))`. The detail string exposes internal exception text to clients (information disclosure) and masks real 5xx observability signals in APM tooling. | Catch only the specific expected types (`ValueError`, `AlloyMismatchError`, `HTTPException` re-raise). Let unexpected exceptions propagate to FastAPI's default 500 handler which logs with full traceback and returns a generic body. Most of these blocks are unnecessary — the `create_metal_purchase`/`consume_material` ones already have a typed ladder; remove the trailing generic catch. |
| P1 | `src/goldsmith_erp/api/deps.py:15`, `api/routers/auth.py:26`, `main.py:148-160` | `OAuth2PasswordBearer(tokenUrl=..., auto_error=False)` + custom cookie logic + WebSocket query-param fallback means JWT extraction happens in three distinct places (`api/deps.get_token_from_cookie_or_header`, `middleware/auth_required._extract_token`, `main._authenticate_websocket`) with subtly different rules — e.g. the middleware accepts an `Authorization` header with any prefix, the deps module only accepts `Bearer ` via OAuth2PasswordBearer auto-parsing, and the WebSocket helper only accepts the cookie or a `token` query param. | P1 | `src/goldsmith_erp/services/system_monitor.py:316-329` | The tuple of `(name, coroutine)` pairs on line 316 **eagerly creates every coroutine object** (each `NotificationService.check_X(db)` call runs up to the first `await` and returns a coroutine). ## Strengths observed

- **Consistent `selectinload()` coverage** across 18 of 31 services for relationship access; `OrderService.get_orders/get_order` is exemplary. N+1 risks are mostly limited to services like `photo_service`, `label_service`, and `import_service` that don't touch relationships — i.e. the 13 services without `selectinload` genuinely don't need it.
- **`transactional(db)` helper in `db/transaction.py`** is the right abstraction and is correctly used by `OrderService` (`create_order`, `update_order`, `delete_order`, `change_location`). It just needs to be adopted uniformly.
- **Pub/sub layer's retry + circuit-breaker semantics** in `core/pubsub.publish_event` (3 retries, exponential backoff, never raises) is well-designed and the right choice for fire-and-forget events.
- **Scanner router's boundary discipline** (`api/routers/scanner.py`) is a model for the rest: single `@require_permission`, explicit "never trust `user_id` from body", explicit ORM→Pydantic mapper instead of `from_attributes`, and a well-documented contract in the module docstring.
- **JWT refresh grace window** in `core/security.decode_token_allowing_grace_window` is a thoughtful UX detail — 5-minute grace for recently-expired tokens avoids forcing re-login on flaky networks without extending the effective token lifetime.
- **Fail-fast production config validation** in `core/config._check_encryption_key` and `_check_anonymization_salt` — refuses to start in non-DEBUG mode without the required secrets. This is correct 12-Factor discipline.

## Open questions / areas needing follow-up

- Is `/users/register` intentionally public (hackathon-style open signup) or an MVP oversight? The default role resolution needs a ground-truth decision from the product owner before flagging it as a hard P0.
- `_safe_publish` in `time_tracking_service.py` claims A5.5 in-app-notification-on-pubsub-failure but the code path cannot fire (see P1 finding). Need to confirm with the author whether `publish_event` is supposed to raise on retry exhaustion, or whether the contract doc is aspirational. The DPIA / compliance agent may also have an opinion.
- The `_authenticate_websocket` helper reads `token` from `websocket.query_params`. Query params are logged by reverse proxies by default (nginx `access.log`). A short-lived one-time WS token derived from the JWT (stored in Redis with 60 s TTL) would remove the JWT-in-URL footprint. Worth checking what the ingress config (Caddy/nginx) is doing on the deployed environment before flagging.
- `photo_service.upload_photo` reads `max_bytes + 1` from the `UploadFile`. Starlette's `UploadFile.read(n)` will happily exceed the client's advertised Content-Length if the body is chunked — need to verify the 10 MB middleware cap in `main.py::RequestSizeLimitMiddleware` is effective when `Transfer-Encoding: chunked` is used.
- The `@require_permission` decorator + FastAPI signature inspection compatibility: confirmed working via `functools.wraps`, but I did not run the test suite to verify every route's OpenAPI schema still surfaces the `current_user` parameter correctly. A 5-minute check (open `/docs` and look for any endpoint whose auth requirement silently disappeared) would close this out.
