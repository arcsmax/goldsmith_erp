## Strengths observed

- **Transactional helper `db/transaction.py:13-54`** (async context manager with auto-rollback, structured error logging) is used by `order_service.py:179` â€” exactly the pattern CLAUDE.md asks for. Events are published AFTER the `async with transactional(db):` block exits successfully (order_service.py:213, :431, :542). That is the right ordering.
- **`selectinload` discipline is strong**: 119 call sites across services, covering all hot relationships (orders.customer, orders.materials, orders.gemstones, time_entries.activity/user/order/interruptions/photos, valuations.order/customer/creator, etc.). Very few bare relationship accesses.
- **Migration chain is linear, no branches, no merges** â€” easy to reason about.
- **FK `ondelete` semantics are deliberate**: the `20260420_h9` migration plus its docstring table documents 11 RESTRICT FKs, 3 CASCADE (user-owned ephemera), and 9 SET NULL (audit rows). This is the clearest FK policy documentation I've seen in a code review.
- **Idempotent migration helpers** (`db/migration_helpers.py`) let fresh DBs (seeded via `Base.metadata.create_all()`) and legacy DBs run the same migration code path. Subtle and well-designed.
- **Redis pubsub uses context-manager `get_redis_client()`** (CLAUDE.md requirement), with connection pool reuse, retry-with-backoff, and caller never sees exceptions on Redis outage. Channel cleanup (`unsubscribe` + `close`) is in a `finally` block.
- **Partitioning of `scan_logs`** â€” thoughtful PG-only feature with a clean SQLite fallback, composite PK for the partition key, partial index on idempotency_key acknowledging the partition-vs-uniqueness trade-off.
- **Sentinel user `id=0` for GDPR anonymization** is well thought through: explicit INSERT with `ON CONFLICT (id) DO NOTHING`, bcrypt-invalid password `'!'`, VIEWER role, `is_active=FALSE`.

## Open questions / areas needing follow-up

1. **Production PII encryption status** â€” Is `Customer.email` currently stored plaintext or ciphertext in production? The `.ilike()` usage in `search_customers` strongly implies plaintext. If so, this is a live GDPR exposure that needs a one-time encryption migration.
2. **`v1_initial`'s reliance on live `Base.metadata.create_all()`** â€” when someone months from now adds a new table + migration but forgets to add the table to `POST_V1_TABLES` in `v1_initial`, future fresh-DB bootstraps will create it via ORM then fail when the new migration tries to `CREATE TABLE` again. Consider committing to standard `op.create_table` migrations going forward and freezing `v1_initial` as a literal snapshot (use `alembic upgrade head` from a known DB dump rather than metadata-driven).
3. **Partition extension job** â€” who owns the cron that adds `scan_logs_YYYY_MM` past 2027-04? Nothing in the repo creates it. Recommend landing a scheduled `scripts/rotate_scan_partitions.py` before first rows write to `scan_logs_default`.
4. **Float-to-Numeric migration plan** â€” this is the single highest-impact finding. Needs: (a) precision map per column, (b) batch UPDATE with explicit CAST to avoid round-trip rounding, (c) frontend coordination (JSON numbers with 2-digit precision), (d) invoice reconciliation sanity check on existing data.
5. **Test coverage against PG** â€” the SQLite-vs-PG schema drift on UUIDs, partitioning, JSONB defaults means SQLite tests can miss a real class of bugs. Is there a nightly CI run against a Postgres container? If not, add one.
