# Review — GDPR / Privacy / Data Protection  (2026-04-23, HEAD 1feae6d)

## Scope reviewed

Reviewed the committed HEAD (commit `1feae6d`) of `/Users/maxbook/Documents/Github/Anne/goldsmith_erp` for GDPR compliance.  In scope: `src/goldsmith_erp/db/models.py` (schema of record), customer / user / valuation / scrap-gold / invoice / photo / comment / scanner routers and their services, `core/encryption.py`, `core/permissions.py`, `middleware/audit_logging.py`, the two GDPR migrations (`20260406_review`, `20260417_anonymize_user`), the `FileErasureService`, `CustomerService.scrub_customer_pii`, `UserService.anonymize_user`, `scripts/gdpr-cleanup.sh`, and `docs/GDPR_COMPLIANCE.md`.  Out of scope (handed to the security agent): authentication, JWT storage, XSS, injection, CORS, CSRF.

The codebase has several well-engineered GDPR features (file-erasure sweep with path-traversal guard, user anonymisation with HMAC audit tokens, GDPR request lifecycle tracking, PII scrub targets as a declarative registry).  It also has a set of load-bearing gaps that together make it **not compliant with the project's own `CLAUDE.md` "Data Privacy Rules (CRITICAL)"** — primarily around PII encryption scope, financial-data access control, retention enforcement, and a dead middleware.

## Findings

| Severity | File:line | Issue | Fix recommendation |
|---|---|---|---|
| **P0** | `src/goldsmith_erp/db/models.py:221-226`, `services/customer_service.py:267` | **Names and email of customers are stored in plaintext.**  CLAUDE.md explicitly lists *"Names, addresses, phone numbers, email → MUST be encrypted at rest (EncryptedString type)"* as non-negotiable.  `PII_FIELDS = ["phone", "mobile", "street", "city", "postal_code"]` omits `first_name`, `last_name`, `company_name`, and `email`.  Email is additionally indexed and unique-constrained, which makes any future encryption migration harder. | Either (a) implement a real SQLAlchemy `TypeDecorator` named `EncryptedString` and attach it to every PII column, or (b) at minimum extend `PII_FIELDS` to `["first_name", "last_name", "company_name", "email", "phone", "mobile", "street", "city", "postal_code", "birthday"]`, write a backfill migration, and replace the unique-email-index with a deterministic blind-index (HMAC) column. |
| **P0** | `src/goldsmith_erp/db/models.py:2039`, `api/routers/valuations.py` | **Insurance valuation (`appraised_value`) is stored in plaintext.**  CLAUDE.md: *"Valuation data MUST be encrypted at rest."*  The class docstring at `db/models.py:1996-1998` acknowledges the sensitivity ("financial data"), yet the column is a plain `Float`. | Add `appraised_value_encrypted` (String) + `appraised_value_hmac` (for searches), route writes/reads through `core/encryption.py`, and deprecate the plaintext column with a follow-up migration. |
| **P0** | `src/goldsmith_erp/services/customer_service.py:279-315` | **PII encryption is best-effort — exceptions are swallowed.**  `_encrypt_pii` catches `Exception` and keeps the plaintext (lines 291-295); `_decrypt_pii` does the same.  This means a misconfigured `ENCRYPTION_KEY` silently persists plaintext PII into PostgreSQL.  CLAUDE.md "fail loudly" principle + the Decision-Making Hierarchy ("security > correctness > performance > convenience") are violated. | Raise `EncryptionError` when encryption is required and the key is absent; at the very least log an error tagged `audit=True`.  Add a startup health check that asserts `ENCRYPTION_KEY` is set in any environment that has real customer rows. |
| **P0** | `src/goldsmith_erp/core/permissions.py:175-199`, `models/order.py:294-344`, `api/routers/orders.py:20-30` | **VIEWERs see all financial fields on orders.**  `UserRole.VIEWER` has `ORDER_VIEW`.  `OrderRead` contains `price`, `material_cost_calculated`, `material_cost_override`, `labor_cost`, `hourly_rate`, `profit_margin_percent`, `calculated_price` (lines 313-323).  `GET /orders/` returns them unredacted. | **P0** | `api/routers/invoices.py`, `api/routers/valuations.py`, `api/routers/scrap_gold.py` — all GET handlers | **Financial data reads are not audit-logged.**  CLAUDE.md: *"All financial data access MUST be audit-logged."*  The three financial routers enforce `*_VIEW` permissions but no `CustomerAuditLog` or structured-log row is written when an invoice / valuation / scrap-gold record is fetched.  Only `scripts/gdpr-cleanup.sh` and `/customers/{id}/export` emit audit rows. | Add a dependency `audit_access(entity, entity_id)` wired into every financial-read endpoint.  Minimum: write a row with `action="financial_read"`, `user_id`, `entity`, `entity_id`, `ip`, `timestamp`.  Consider a service-layer decorator so future endpoints cannot forget it. |
| **P0** | `src/goldsmith_erp/main.py:97-113`, `middleware/audit_logging.py` | **`AuditLoggingMiddleware` is implemented but never registered.**  The file documents itself as writing `CustomerAuditLog` rows for every access to `/api/v1/customers` (lines 74-86), yet `main.py` only adds `AuthRequiredMiddleware`, `RequestSizeLimitMiddleware`, `RequestLoggingMiddleware`, `RequestMetricsMiddleware`, `CORSMiddleware`, `SecurityHeadersMiddleware`.  GDPR Art. 30 (Records of processing activities) relies on this data. | Register it in `main.py` alongside the other middleware.  Verify via an integration test that a `GET /api/v1/customers/1` produces a `CustomerAuditLog` row with action=`read`. |
| **P0** | `src/goldsmith_erp/db/repositories/customer.py:464-535` | **Consent management code references columns that do not exist.**  `update_consent` sets `consent_marketing`, `email_communication_consent`, `phone_communication_consent`, `sms_communication_consent`, `data_processing_consent`, `consent_date`, `consent_version`, `consent_ip_address`, `consent_method` — none of which are columns on `Customer` (see `db/models.py:214-276`).  Any call will raise `AttributeError` at runtime. ## Strengths observed

- `services/file_erasure_service.py` is genuinely good privacy engineering — declarative targets, path-traversal guard, NOT-NULL sentinel, per-file audit.
- `services/user_service.anonymize_user` uses an HMAC tracking token in logs rather than the raw user id — exactly what CLAUDE.md asks for.
- `services/customer_service.scrub_customer_pii` with the declarative `SCRUBBABLE_FIELDS` registry is easy to audit and easy to extend.
- `api/routers/customer_portal.py` is an exemplar of Privacy-by-Default: the module docstring (lines 1-11) says "No PII, no financial data, and no design details are ever returned" and the rate-limit + token-TTL pattern backs it up.
- `services/email_service.py:179,208` anonymises recipient addresses in structured logs.
- `GDPRRequest` lifecycle (PENDING → completed / PARTIAL_FILE_ERASURE / FAILED) in `api/routers/customers.py:460-707` is careful and idempotent.
- Scanner QR payload (`ORDER:{id}`, `REP:{id}`, `SCRAP:{id}` — `services/label_service.py:226`) carries no PII.

## Open questions / areas needing follow-up

1. **Tenant isolation** — the `users.tenant_id` forward-compat column is nullable and unused.  If V1.2 multi-tenancy ships before a migration backfills every row, cross-tenant leakage is guaranteed.  Flag this for the V1.2 planning.
2. **Encryption key rotation** — the encryption service uses a single Fernet key (`settings.ENCRYPTION_KEY`).  There is no key-version column on any encrypted field and no key-rotation runbook.  Ask the ops lead whether rotation is a V1.1 gate.
3. **ML data and anonymisation** — `ml/feature_engineering.py` and `services/ml_data_service.py` export training features from live tables.  Do those exports strip customer-identifying fields?  Worth a dedicated read by the ML reviewer (out of scope here).
4. **CSV import path** — `services/import_service.py` accepts `birthday` and `notes` for bulk customer import.  No consent capture, no legal-basis column.  If this endpoint is used in production, it's a GDPR violation regardless of the fixes above.
5. **`docs/GDPR_COMPLIANCE.md` status** — the document header says "System is currently NOT GDPR compliant" (line 14) from 2025-11-06.  The V1.1 work has closed some gaps (file erasure, user anonymisation, PII scrub) but has not revised the doc.  Confirm whether this is a code-ship blocker for V1.0.
6. **`CustomerAuditLog.details` JSON contents** — the dry-run and file-erasure paths embed the whole `FileErasureResult.as_dict()` in `details`, which includes resolved filesystem paths.  Those paths can re-identify a deleted customer if the storage layout is customer-id based.  Worth a targeted read.
