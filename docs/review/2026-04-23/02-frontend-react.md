# Review — Frontend React Quality  (2026-04-23, HEAD 1feae6d)

## Scope reviewed

Audited HEAD of `frontend/src/` (~153 files) with focus on: entry & routing (`main.tsx`, `App.tsx`), the Axios client + token-refresh interceptor, all five React contexts (Auth, Order, TimeTracking, Scanner, Toast, Theme), the `useWebSocket` / `useTheme` / `useHidBurstDetection` / `useMetalTypes` hooks, `MainLayout`, `ProtectedRoute`, the Slice-8/10/11 scanner stack (`QrCameraScanner` **committed** version, `ScanOverlay`, `ScanFab`, `QuickActionModalV2`, `ActionHandlers`), both timer widgets, `GlobalSearch`, `NotificationBell`, `AuthenticatedImage`, `OrderList`, several pages (`LoginPage`, `OrderDetailPage`, `CustomerPortalPage`, `InvoicesPage`), `types.ts`, `tsconfig.json`, `vite.config.ts`, and `package.json`. Cross-cutting greps covered `any`/`as any`/non-null `!`, unsafe-HTML-injection props, `div onClick`, `window.prompt`/`confirm`, raw `fetch()`, empty-dep `useEffect`, `react-hooks/exhaustive-deps` disables, `localStorage` usage, and `ErrorBoundary`. Note: authentication migrated from `localStorage(access_token)` to HttpOnly cookies (`withCredentials: true`) — token-in-LS is **no longer** a P0. No occurrences of the unsafe-HTML-injection prop were found in the tree — good.

## Strengths observed

- **Scanner code quality is genuinely high.** `QrCameraScanner`, `ScanOverlay`, `QuickActionModalV2`, `ActionHandlers` demonstrate careful stream-lifecycle cleanup (the 2026-04-18 probe-release fix at `QrCameraScanner.tsx:193-202` is correctly documented in the source and tests), explicit focus trap + Esc-to-close (WCAG SC 2.4.3), `prefers-reduced-motion` respect, German copy discipline, and deliberate separation between rendering and side-effects.
- **Token refresh interceptor is correct.** `client.ts:32-134` handles the "refresh storm" case with a queue, avoids infinite loops on the refresh endpoint itself, and broadcasts `auth:session-expired` so React state updates follow. Cookie-based auth (`withCredentials: true`) is a healthy improvement over the documented-but-stale localStorage JWT pattern.
- **Test coverage on Slice 8–12 is real.** 13 test files totalling ~4.5k LOC exercise scanner routing, HID burst detection, bench-scanner listener, ActionHandlers, ScanOverlay, QuickActionModalV2, ScannerContext — and they mock the heavy `@yudiel/react-qr-scanner` vendor cleanly so tests don't pull camera code into jsdom/happy-dom.
- **Code-splitting is deliberate.** All 22 pages are `React.lazy`-imported in `App.tsx`; scanner vendor module is dynamic-imported inside `QrCameraScanner.tsx:81`; a dedicated CI gate (`.github/workflows/bundle-size-gate.yml`) enforces a 250 KB scanner chunk ceiling on every PR.
- **A11y discipline is above average for the codebase age.** `aria-label`, `aria-expanded`, `aria-controls`, `aria-live`, `role="dialog"`/`"alert"`/`"combobox"` used in the new components; `NotificationBell` does colorblind-safe icons alongside color; `GlobalSearch` implements keyboard navigation with `ArrowUp`/`ArrowDown`/`Enter`/`Escape`.

## Open questions / areas needing follow-up

- **Should the React Context layer migrate to Zustand?** CLAUDE.md references the recommendation. My take: contexts are not the problem today — the problem is the missing `useMemo` on provider `value` (P2) and the doubled WebSocket (P1). Fix those first; a Zustand migration is a separate call justified by per-slice selector perf.
- **Is `ActiveTimerWidget` actually used anywhere?** It looks orphaned. Confirming would unlock a ~460-LOC deletion (P2).
- **Does the backend actually send `SameSite=Strict` + a CSRF token?** P0 depends on this. Needs security-agent confirmation.
- **Intentional `// eslint-disable react-hooks/exhaustive-deps` in `QrCameraScanner.tsx:277`, `useMetalTypes.ts:119`, `useWebSocket.ts:139`, `NotificationBell.tsx:150`, `TimeTrackingContext.tsx:298` (implicit via empty deps).** Each is defended in a comment, but four of five could be fixed by including the referenced memoized callbacks in deps — confirm team convention.
- **Direct `window.location.href = '/login'` in interceptor (`client.ts:86, 129`).** Bypasses react-router, loses `location.state` (e.g. `returnTo`). Is there a product decision to always land on a clean `/login`, or is the `from` route meant to be preserved for post-login redirect?
- **`types.ts` is 1069 lines of hand-maintained types mirroring backend Pydantic schemas.** No codegen. Several types disagree subtly with their Pydantic counterparts (e.g. `NotificationBell.tsx:139` reads `message.action === 'create'` but `WebSocketMessage` and `Notification` types have no `action` field). Any plan to adopt `openapi-typescript` or `orval`?
