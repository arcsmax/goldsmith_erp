# Review — Security Audit  (2026-04-23, HEAD 1feae6d)

## Scope reviewed
Committed HEAD (`1feae6d`) of the Goldsmith ERP backend and frontend. Focus:
JWT lifecycle (`core/security.py`, `core/config.py`, `api/routers/auth.py`,
`middleware/auth_required.py`, `api/deps.py`), RBAC (`core/permissions.py`
plus the shadow implementation in `api/deps.py`), every router under
`api/routers/` (32 modules) grep'd for unauthenticated routes, input
validation across Pydantic request models, file-upload surfaces (photos,
materials, scrap-gold, imports), SQL handling (grep for `text()`, f-string
SQL, `eval`/`exec`/`subprocess`), transport/CORS/CSP configuration
(`main.py`, `middleware/security_headers.py`), container and compose files
(`Containerfile`, `Dockerfile`, `podman-compose.yml`,
`podman-compose.prod.yml`, `docker-compose.yml`), secrets hygiene
(`.env.example`, `.gitleaks.toml`, committed `.env` check), WebSocket auth
(`main.py`), scanner/QR surface (`api/routers/scanner.py`,
`services/scanner_service.py`, `models/scanner.py`), frontend auth
(`frontend/src/api/client.ts`, `contexts/AuthContext.tsx`), XSS risk
(grep for unsafe-HTML-render patterns), and PII logging patterns.
GDPR/data-protection lifecycle rules are left to the compliance agent;
this review flags secrets, auth flaws, injection, and hardening gaps
only.

## Threat model context
Internal workshop ERP, typical deployment: a few workstations, a tablet,
and a bench USB-HID scanner on a LAN behind a reverse proxy. Plausible
threat actors (highest impact first): (1) an outside attacker who has
either a low-privileged foothold on the LAN or reached the backend via
an exposed port / reverse-proxy misconfig; (2) a disgruntled VIEWER /
GOLDSMITH role inside the workshop escalating to financial data or
insurance-valuation PDFs; (3) a customer using the public self-service
portal trying to enumerate other customers' orders; (4) a supply-chain
attacker via dependency (covered by dedicated CVE agent). Crown jewels:
customer PII (encrypted via Fernet when `ENCRYPTION_KEY` is set),
insurance valuations (financial, access-controlled at router layer),
design IP, and pricing/scrap-gold financial fields. Exposure: backend
publishes `/docs` and `/openapi.json` publicly; customer portal endpoint
is public and rate-limited; login endpoint is public and rate-limited.

| **P1** | `src/goldsmith_erp/api/routers/customer_portal.py:360-380` | Public `/portal/lookup` is rate-limited to 10 req/min per IP. Once a valid id is found, the email-address check is the only gate, and common leaked-email lists make the brute-force feasible. A successful pair returns a lookup token (`secrets.token_urlsafe(32)`, 1-hour TTL) that is directly useful as a share link without further gate. | Replace numeric IDs in public references with a random opaque token (e.g. `ORD-` + 12 urlsafe chars) stored in a dedicated column. ## Strengths observed

* JWT signing is pinned to HS256 in `core/security.py:12`; the `jwt.decode` calls in `api/deps.py:67`, `middleware/auth_required.py:74`, and `main.py:156` all pass `algorithms=[ALGORITHM]`, defeating "alg: none" attacks.
* `SECRET_KEY` validator in `core/config.py:184-238` blocks the documented insecure defaults, enforces >=32 chars, and warns on low entropy. It even catches the value that the dev compose file defaults to — so the backend refuses to start with a weak key rather than silently using one.
* `Containerfile` runs as non-root (`appuser:1000`), uses a healthcheck based on Python stdlib (no curl needed), and builds off `python:3.11-slim`.
* Photo uploads validate by magic bytes, not filename or `Content-Type` (`services/photo_service.py:54-63`). Same pattern in `scrap_gold.py:144` and `materials.py:316`.
* CSV import uses `decode("utf-8-sig")` and collects per-row errors rather than aborting, but enforces a hard 5 MB cap (`imports.py:32`, `:108`).
* The scanner Pydantic schemas (`models/scanner.py`) use `StrictRequestBase` with `extra="forbid"`, reject control chars in `raw_payload`, cap it at 500 chars, strict-whitelist the `ScanContext`, and explicitly refuse a `user_id` in the body — the router also re-asserts that `user_id` comes from the JWT (`api/routers/scanner.py:147-148`).
* `get_real_ip` in `middleware/audit_logging.py:51-71` and `middleware/rate_limiting.py:44-64` correctly only trusts `X-Forwarded-For` when the direct TCP peer is loopback or RFC-1918 — clean defence against header-spoof attacks.
* No raw SQL injection surface: `git grep "execute(f"` and `git grep "text(\""` return zero user-input interpolations; the 4 `text()` usages are constant health-check queries and Alembic server defaults.
* No unsafe-HTML render calls anywhere in `frontend/src/`. Access tokens are **not** stored in `localStorage` (only the `user` metadata object) — auth relies on the HttpOnly cookie, with a header fallback only for back-compat paths.
* Refresh flow with 5-minute grace window (`core/security.py:19`) is well-scoped and logged; rate-limited at 10/min.
* Notification WebSocket (`main.py:172`) pins `authenticated_user_id == user_id` — a user cannot subscribe to another user's channel.

## Open questions / areas needing follow-up

1. **Audit-log completeness.** `AuditLoggingMiddleware` is defined but unregistered. Does manual `AuditLogService` coverage hit every financial/PII endpoint? Needs a dedicated sweep of routers that mutate `invoices`, `quotes`, `valuation_certificates`, `scrap_gold`, `customers`.
2. **Password-change token invalidation.** Is there a test asserting that changing a password invalidates outstanding JWTs? The 8-day lifetime amplifies the consequences if not.
3. **Dependency freshness.** Not inspected in detail — handed to the CVE agent. Worth noting: `jose` (python-jose) has had a maintenance pause in 2024-2025; if the project is still on it, consider migrating to `PyJWT`.
4. **MFA.** `PUBLIC_PREFIXES` contains `f"{settings.API_V1_STR}/auth/mfa"` — suggests a planned MFA router that does not yet exist. The whitelist should not pre-permit a path that has no handler; remove until the router lands or a malicious route with that prefix would go un-auth'd if mistakenly mounted.
5. **Reverse-proxy assumptions.** Production deployment documentation (not reviewed exhaustively here) should mandate `X-Forwarded-Proto` so `secure` cookies are actually sent, and a LAN-only bind (`127.0.0.1:8000` behind nginx) rather than the `0.0.0.0:8000` publish in `podman-compose.prod.yml:74`.
6. **`.dockerignore` contents.** Not inspected beyond existence — verify `.env`, `.env.*` (except `.env.example`), `uploads/`, `tests/*.db`, `.serena/`, `.claude/` are all excluded from image builds.
7. **PUT /api/v1/theme logo upload path.** If a future slice adds "upload a logo" rather than URL, revisit the same magic-byte + origin-pinning guarantees as photos.
